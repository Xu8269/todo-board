# 个人实训总结报告

## 一、项目概况

本次实训我独立完成了「TaskPlanner 任务看板系统」的开发。项目基于 Next.js 16 (App Router) + TypeScript + TailwindCSS + Mongoose + MongoDB 技术栈，实现了三栏看板（待办/进行中/已完成）的核心功能，并附加了仪表盘日历、深色模式切换、手绘鱼缸互动等特色功能。

项目代码托管在 GitHub，共 26 次 Git 提交记录，部署至 Railway 平台但受限于国内网络环境无法公网访问。项目包含 3 个前端路由页面、11 个后端 API 接口，所有 API 均使用 try-catch 错误处理，代码结构符合 Next.js 规范。

## 二、遇到的最大困难及解决过程

### 2.1 深色模式的 hydration 错误

在实现深色模式切换功能时，遇到了严重的 React Hydration 错误。原因是 Next.js 服务端渲染时无法访问 localStorage，导致服务端渲染亮色模式，而客户端首次渲染时通过 useState 惰性初始化读取 localStorage 后直接使用深色模式，两边渲染结果不一致，React 报错并拒绝修复差异。

解决过程经历了三次迭代：第一次尝试在 ThemeContext 中用 mounted 状态延迟渲染，发现 isDark 的惰性初始化在客户端第一次渲染时已经返回了深色值，单纯延迟渲染无法解决。第二次方案改为移除惰性初始化，全部使用 useState(false) 默认值，但出现页面闪烁。最终采用了两层防御策略——全局布局的 `<html>` 标签添加 `suppressHydrationWarning`，同时在每个使用主题色的组件中增加 mounted 判断，hydration 完成前使用亮色 fallback 值。这个方案彻底解决了错误，代价是首次加载有轻微闪烁。

### 2.2 部署后的域名访问问题

项目在 Railway 上部署成功后，日志显示应用运行正常（`Ready in 87ms`），Railway 也生成了 `todo-board-production-223f.up.railway.app` 域名，并通过 TXT 记录验证了自定义域名 `www.todoboard.top` 的 DNS 配置。但实际访问时浏览器报连接超时。

排查过程从检查 Railway 服务日志（确认应用正常运行，监听 8080 端口）、验证域名 DNS 记录（在 Namesilo 配置 CNAME 指向 Railway、通过 Railway 的 TXT 验证）、到使用 whatsmydns.net 确认全球 DNS 传播完成。最终确认是 Railway 的美国服务器节点 IP 在中国大陆被限制访问，这个网络层面的限制无法通过代码解决。作为替代方案，以本地录屏作为功能演示。

### 2.3 摸鱼鱼缸的性能与交互优化

手绘鱼缸功能经历了三次重构：最初是 Canvas 动画自动游动，用户只能投喂；第二次改为用户绘制 + 投放鱼缸，但鱼的身体套用了固定模板；第三次实现了所画即所得的完整交互——用户画的每个像素都直接作为鱼的身体纹理。

核心技术挑战有两个：一是如何让鱼的游动自然真实，最终通过缓动转向算法（目标向量平滑插值）+ 鱼尾摆动频率随速度自适应 + 周期性随机游走路径实现；二是将绘制的 base64 图案持久化到 MongoDB，页面刷新后鱼不会消失。

## 三、对 AI 辅助编程的心得体会

### 3.1 AI 作为编程伙伴而非替代者

在本次实训中，AI（ChatGPT）承担了部分代码生成、Bug 定位、方案设计等角色。但 AI 不是简单的代码生成器——它更像一个有经验的编程伙伴。AI 生成的代码需要人工理解、验证和调整。特别是在处理 Next.js 的 Server Action、React Hydration 机制等框架特有概念时，AI 给出的方案有时需要多次迭代才能正确适配。

### 3.2 提问质量决定回答质量

一个核心体会：把大问题拆成小问题逐步追问，比一次性提笼统问题效果好得多。例如深色模式，没有一次性问"怎么实现深色模式"，而是拆解为创建 ThemeContext → 组件适配 → 修复 hydration 错误三步。每步的 AI 输出更精准，也更容易理解和整合。这个经验不仅适用于编程，本质上是一种高效的问题解决思维。

### 3.3 识别 AI 输出的局限性

AI 在以下方面有明显局限：无法处理网络基础设施问题（如 Railway 域名被屏蔽）、对特定框架版本的 API 变化不够敏感（例如 Next.js 16 的某些特性与 14 不同）、生成的代码有时缺少边界情况处理。这些都需要开发者主动识别和补全，说明 AI 辅助编程的实质是"人主导、AI 辅助"。

## 四、个人贡献

- **项目架构设计**：确定了 Next.js App Router + MongoDB 的技术方案，设计了 3 个前端路由（/ /tasks /tasks/new）和 11 个 RESTful API 端点（tasks CRUD + stats + fish）的接口规范
- **核心功能实现**：独立完成了任务 CRUD、三栏看板、优先级排序（聚合管道）、逾期标红、深色模式（ThemeContext + CSS 变量）、日历视图（截止日期标黄）、一键清理、删除确认弹窗等全部功能
- **鱼缸系统**：设计了 Canvas 动画引擎 + 手绘组件 + MongoDB 持久化的完整方案，鱼的游动采用缓动转向 + 随机游走算法
- **工程化实践**：环境变量管理配置（MONGODB_URI）、try-catch 覆盖 11 个 API 异常处理、Git 提交规范化（feat/fix/docs 前缀）、CSS 变量管理主题色
- **代码审查与重构**：通过 AI Code Review 发现 7 个潜在问题，包括 hydration 错误、状态同步不一致、缺少 mounted 保护等，全部修复
- **文档建设**：编写了 README.md（项目介绍+技术栈+运行指南+API 文档）、API.md（11 接口说明）、prompt_log.md（16 条 Prompt 记录）

## 五、总结

本次实训让我完整经历了一个全栈项目的开发流程——从需求分析、技术选型，到编码实现、测试验证、部署上线的完整闭环。过程中遇到了不少真实工程问题：Next.js 的 Hydration 机制、MongoDB 聚合管道排序、Canvas 动画性能优化、域名 DNS 配置、CI/CD 部署流程等。每一个问题的解决过程都加深了对技术栈底层机制的理解。

更重要的是，通过深度使用 AI 辅助编程，体会到"人机协作"的实质——AI 擅长代码生成和模式匹配，但架构决策、问题拆解、质量检查仍需开发者主导。这些经验对未来工程实践具有直接参考价值。
